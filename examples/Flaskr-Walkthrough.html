What actually happens, from first visit to a published post.

Flaskr is the official Flask tutorial blog: an application factory, two blueprints, a per-request SQLite connection, and session-cookie auth. This traces every route from tap to database — grounded in the source, line by line.

00Architecture

03Log in & the session

flaskr/auth.py:84–109 · route /auth/login · methods GET, POST

A POST looks the user up by name, verifies the hash, and — on success — stores their id in a fresh session cookie.

1
lookupSELECT * FROM user WHERE username = ? → .fetchone() — auth.py:92–94
2
verifyif user is None → "Incorrect username."; elif not check_password_hash(user["password"], password) → "Incorrect password." — auth.py:96–99
3
sessionsession.clear() then session["user_id"] = user["id"] — auth.py:103–104. The cleared-then-set pattern prevents session fixation.
4
redirectredirect(url_for("index")) — auth.py:105
Why a session id, not a token

Flask signs the session cookie with SECRET_KEY. Only user_id lives in it; the full user row is re-loaded from the database on every request by load_logged_in_user (§08). There is no JWT, no server-side session store — the signed cookie is the session.

Username enumerationSeparate "Incorrect username." vs "Incorrect password." messages (auth.py:97,99) reveal whether an account exists. See §11.

04The index & the auth gate

flaskr/blog.py:16–25 · route / (aliased index) · public

The landing page lists every post, newest first, joining in each author's name:

SELECT p.id, title, body, created, author_id, username
  FROM post p JOIN user u ON p.author_id = u.id
 ORDER BY created DESC

Rendered by blog/index.html: the New link shows only {% if g.user %}, and each post's Edit link shows only {% if g.user['id'] == post['author_id'] %} — the UI mirrors the server-side ownership rule.

The gate — login_required

auth.py:19–29 — a decorator wrapping the write views. If g.user is None it redirects to auth.login; otherwise it calls the view. This is the single choke point protecting create, update, and delete.

06Edit & delete

flaskr/blog.py:86–125 · /<int:id>/update (GET, POST) & /<int:id>/delete (POST) · both @login_required

Both routes funnel authorization through one helper, get_post (blog.py:28–57):

1
fetchSELECT … FROM post p JOIN user u … WHERE p.id = ? → .fetchone() — blog.py:43–49
2
existsif post is None: abort(404, "Post id {id} doesn't exist.") — blog.py:51–52
3
ownif check_author and post["author_id"] != g.user["id"]: abort(403) — blog.py:54–55

Update calls get_post(id) first (:90), then on POST runs UPDATE post SET title=?, body=? WHERE id=? (:105). Delete is POST-only, calls get_post(id) for its 404/403 side effects (:121), then DELETE FROM post WHERE id=? (:123). The update.html delete button carries a JS confirm('Are you sure?') (:17).

One rule, one placeOwnership lives solely in get_post's abort(403). Both mutating routes share it, so the authorization surface is a single line — easy to audit, hard to bypass.

07Log out

flaskr/auth.py:112–116 · route /auth/logout

session.clear() drops user_id, then redirect(url_for("index")). On the next request load_logged_in_user sees no user_id and sets g.user = None — the nav reverts to Register / Log In.

Logout is a GET/auth/logout takes no methods arg, so it is GET-only and has no CSRF token — a third-party page can force a logout. Low impact, but see §11.
